## 二、核心经验

### 经验 1: Polars `.over()` 的惰性求值陷阱

**教训**: `select(rank(rolling_expr).over('date'))` 会在每个 date 分组内重算 `rolling_expr`，导致 NaN。

**修复**: 用 `with_columns()` 物化中间结果。

**规则**: 涉及窗口操作 + 分组操作时，必须先物化再分组。

### 经验 2: LLM 对 Polars 表达式的理解不足

**教训**: LLM 看到比较运算符会生成 Python `if`，但 Polars 不支持。

**修复**: prompt 中加 "DO NOT use if" + `_validate_safety` 正则拦截。

**规则**: LLM 生成代码时优先考虑"看起来正确"而非"运行正确"。对 Polars 特有约束需要显式禁止。

### 经验 3: ReAct > 1-shot

**教训**: 1-shot LLM 调用无法修复拼写错误（如 `.out()` vs `.over()`）。

**修复**: ReAct 循环 + OBSERVE 反馈，让 LLM 自修复。

**规则**: LLM 代码生成必须有重试机制。

### 经验 4: 二值因子需要特殊处理

**教训**: 常数/二值因子导致 Pearson IC 方差为零 → NaN。

**修复**: 自动检测 + 加 1e-7 noise。

**规则**: IC 计算前需检查因子方差。

### 经验 5: Prompt 复杂度与 LLM 遵守度成反比

**教训**: prompt 规则越多，LLM 越容易违反某条规则。

**修复**: 精简为 3 条核心规则 + 开头 "DO NOT" 列表。

**规则**: prompt 规则控制在 3-5 条，负面约束比正面指导更有效。

### 经验 6: Git First, Doc First

**教训**: 改动太多，容易丢失上下文。

**修复**: 每个阶段先 commit + 写设计文档。

**规则**: 复杂任务分阶段 commit，每阶段有明确的输入/输出。
