## 1. The 5 Critical Bug Fixes

### 1.1 Tool result feedback loop

**Bug:** `service.py:283-347` ran a single LLM call without feeding tool results back. Multi-tool
flows stopped at the first tool call.

**Fix:** New `ChatBase.aask_with_tools()` async iterator. Reason → action → observe loop with
`max_iterations=4` (default `DEFAULT_MAX_CHAT_ITERATIONS`). Tool results appended as messages
for the next LLM call.

### 1.2 `_save_message` silent error swallowing

**Bug:** `except: pass` hid DB write failures.

**Fix:** `logger.error` + `_save_error_count` counter + SSE `save_warning` event emitted on failure.

### 1.3 UUID 8-char collision

**Bug:** `uuid.uuid4()[:8]` gave 32 bits → birthday collision at ~65k rows. Plus `INSERT OR REPLACE`
clobbered concurrent writes.

**Fix:** `uuid.uuid4().hex` (32 hex chars = 128 bits). `INSERT OR IGNORE` for race-safe upserts.

### 1.4 Frontend closure staleness

**Bug:** `done` handler in `AgentChat.tsx:180,259` captured stale state via React closures.

**Fix:** `useRef` mirrors of `currentThinkingRef` / `currentToolCallsRef`; `done` handler reads from refs.

### 1.5 Duplicate DB instances

**Bug:** `AgentService` and `ChatService` each opened their own `ChatDatabase`, breaking transaction
boundaries.

**Fix:** `ChatService.__init__` accepts `chat_db`; `AgentService` injects `app_db.chat` singleton.

## 2. Architecture Merge: ChatBase ↔ ChatService

### 2.1 `ChatBase.aask_with_tools()`

```python
async def aask_with_tools(
    self,
    messages: list[dict],
    tools: list[dict],
    max_iterations: int = 4,
    invoke_tool: Callable | None = None,
) -> AsyncIterator[dict]:
    """Async reason → action → observe loop.
    Returns stream of LLM/tool events.
    """
```

- Custom `invoke_tool` callback lets `ChatService` plug in confirmation flow, text-mode parsing,
  and DB persistence.
- Returns early on `confirmation_required`.

### 2.2 `ChatService` delegation

`ChatService.chat()` and `approve_confirmation_continue()` now delegate to
`ChatBase.aask_with_tools()`. The duplicated `_stream_llm` / `_dispatch_tool_call` loops in
`service.py` were removed.

### 2.3 Text-mode as a hook

`ChatService._stream_preprocess` translates `[TOOL_CALL]...[/TOOL_CALL]` blocks in `content` events
into native `tool_call` events, so text-mode LLMs (no tool_calls field) work transparently.

## 3. MemoryManager Full Integration

### 3.1 System prompt injection

`_build_system_prompt` (now async) composes 6 sections:

1. Role + strategy
2. Wiki context (current page + available pages)
3. User preferences (from `UserPreferenceStore.aall()`)
4. Tool list
5. Current date
6. Relevant history (top-3 from `MemoryIndex.asearch()`)

### 3.2 History via MemoryManager

`_get_or_create_context` (async) prefers `memory.conversation.alist()` over direct DB queries.

### 3.3 Tool result persistence

`_persist_tool_result()` runs after every tool call, best-effort, writing to
`memory.context` (via `ContextStore.aadd()`).

### 3.4 Async wrappers for all 6 stores

| Store | New async methods |
|---|---|
| ConversationStore | `alist`, `aadd`, `asearch` |
| ContextStore | `alist`, `aadd`, `aclear` |
| UserPreferenceStore | `aget`, `aset`, `aall` |
| MemoryIndex | `asearch` |
| KnowledgeStore | unchanged (sync only) |
| ReActStateStore | unchanged (sync only) |

## 4. Reliability Stack

### 4.1 LLM retry

`LLMRetryManager._is_retriable()` (static) — wraps transient errors (5xx, rate limit, network).
`ChatService._llm_stream_with_retry()` retries first-chunk failures up to 3 times (1s/2s/4s backoff).

### 4.2 DB retry

`DBRetryManager._is_retriable()` (static) — wraps SQLite transient errors (database locked, busy).
`_save_message()` uses `_db_retry.call()` for resilience.

### 4.3 Rate limiting

`RateLimitMiddleware` — token-bucket per IP, default 60 req/min, applied to `/api/agent/*` only.
429 + `Retry-After` header on excess.

### 4.4 SSE heartbeat + timeout

- `EventSourceResponse(ping=15)` keeps connection alive through proxies.
- Total stream lifetime 300s (5 min); on timeout a `timeout` event is emitted and stream closes.

### 4.5 Cascade delete + foreign keys

- `delete_chat_session` runs a transaction that deletes messages, tool_calls, context_entries.
- `BaseDatabase._connect()` runs `PRAGMA foreign_keys = ON`.

### 4.6 Architecture fix (bonus)

Removed the L3 → L4 import in `apps/chat/routes.py` (`tool_registry` is now injected via
`set_autoresearch_deps()`).

## 5. Frontend UX

### 5.1 Abort button

`StopCircle` icon replaces the send button during streaming. `handleStop()` flushes buffered text
to message history and calls `abortControllerRef.current.abort()`. `AbortError` does not surface
as an error toast.

### 5.2 Token reset on session change

`handleSelectSession` and `handleNewChat` both reset `tokenEstimate = 0`.

### 5.3 Any-message regenerate

`onRegenerate` is now available on **every** assistant message (not just the last). Clicking it
truncates the conversation to the previous user message and pops the user message content back
into the input box, then re-runs `chatStream`.

New backend endpoint: `POST /api/agent/sessions/{id}/regenerate`.

### 5.4 SSE auto-reconnect

`chatStream` and `approveAndContinue` accept `signal?: AbortSignal`. The internal retry loop:

- Up to 3 retries
- Backoff: 1s → 2s → 4s
- Only retries when no event has been received (avoids duplicate event playback)
- `AbortError` bypasses retry

## 8. Migration

v0.36 is **not backward compatible** with the old broken behavior. However, the API surface is
fully backward compatible. Migration notes:

### 8.1 Caller code (unchanged)

If you previously called `chat_service.chat(session_id, message)` or
`chat_service.approve_confirmation(tool_call_id, approved)`, no change is needed.

### 8.2 New capabilities

- Pass `signal?: AbortSignal` to `chatStream()` for cancellation (optional, defaults to no signal).
- Call `regenerate(session_id, message_id)` to regenerate any assistant message.
- Expect `save_warning`, `timeout` events in addition to the existing set.

### 8.3 Removed

- `apps/chat/agent/service.py::_stream_llm` and `_dispatch_tool_call` (private) — moved into
  `ChatBase.aask_with_tools`.

### 8.4 Deprecation

- `apps/agent/core/{db,service}.py` wrappers from v0.33 — still present, will be removed in v0.37.

## 11. Migration Checklist (for downstream users)

### 11.1 Caller code (no changes needed)

If you previously called:

```python
chat_service.chat(session_id, message)
chat_service.approve_confirmation(tool_call_id, approved=True)
```

**No change required.** The signatures are unchanged.

### 11.2 New capabilities you can adopt

| Capability | How | Reference |
|---|---|---|
| Cancel a running stream | Pass `signal: AbortSignal` to your client wrapper | §5.1 |
| Regenerate any assistant message | Call `regenerate(session_id, message_id)` | §5.3 |
| Listen for `save_warning` events | Handle SSE event type `save_warning` | §1.1 / §1.2 |
| Listen for `timeout` events | Handle SSE event type `timeout` | §4.4 |
| Inspect rate-limit headers | Read `Retry-After` on 429 responses | §4.3 |

### 11.3 Behavioral changes (BREAKING — by design)

| # | Change | Migration |
|---|---|---|
| 1 | UUIDs are now 32 hex chars (was 8) | Update any code that consumed old 8-char IDs |
| 2 | `_save_message` no longer swallows errors | Check for `save_warning` events; monitor logs |
| 3 | Tool calls now iterate 1–4 times (was 1) | Adjust UX if you assumed single-turn |
| 4 | Confirmation flow can interrupt stream | Handle `confirmation_required` event |
| 5 | Foreign keys enforced | Pre-existing inconsistent rows will now fail delete |

### 11.4 Removed (private APIs)

- `apps/chat/agent/service.py::_stream_llm` — moved to `ChatBase.aask_with_tools`
- `apps/chat/agent/service.py::_dispatch_tool_call` — moved to `ChatBase.aask_with_tools`

### 11.5 Deprecated (still works)

- `apps/agent/core/{db,service}.py` wrappers (v0.33). Will be removed in v0.37.

### 11.6 Server / DB prerequisites

- `PRAGMA foreign_keys = ON` is now set automatically. No DB migration required.
- `LLMWIKIFY_RATE_LIMIT_DISABLED=1` env var to disable rate limiting (debugging only).
