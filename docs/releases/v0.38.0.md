## What's New in v0.38.0

v0.38.0 is the **nanobot v0.2.1 borrowing release**: 5 architectural primitives
borrowed from the upstream `nanobot` project land together, plus the bus+WS
wire-up that completes the chat transport story started in Phase 14.

**Before** (v0.37.0): chat events flow directly from `ChatOrchestrator` to the
SSE handler; the WebSocket channel exists but only echoes; no in-process pub/sub;
no provider abstraction; runtime context is inherited globally.

**After** (v0.38.0):
- **Bus-mediated fan-out** ŌĆö every SSE event is mirrored to an in-process
  `MessageBus`, and the WS handler consumes that same bus. Adding a new
  channel is just "subscribe to the bus + translate vocabulary".
- **Real WS chat** ŌĆö `WS /api/ws/agent` `message` routes to `ChatOrchestrator.chat()`
  with session continuity (WS chat_id ŌåÆ backend session_id via `WsSessionMap`).
- **5 new ABCs / primitives** ŌĆö `BusAdapter`, `WsSessionMap`,
  `AgentRunner` ABC, `LLMProviderABC` + `ProviderConfig` + `RetryMode` +
  `ThinkingStyle`, and the Runtime Context Tag wrapper.
- **Real-server integration tests** ŌĆö Phase 17 + 19-C catch a pre-existing
  DB bug (`set_session_metadata` silent failure) and validate the full
  Phase 12-19 wiring on a real `WikiServer` lifecycle.

Highlights:

- **Runtime Context Tag isolation** (Phase 12) ŌĆö `[Runtime Context]ŌĆ”[/Runtime Context]`
  wrapping for per-call tool results, preventing LLM cross-contamination.
- **MessageBus in-process pub/sub** (Phase 13) ŌĆö bounded outbound queue
  with drop+log backpressure; singleton helpers (`get_default_bus()`).
- **WebSocket bidirectional channel** (Phase 14) ŌĆö per-connection out
  queue, fan-out via `WebSocketManager`, auth via `?token=<api_key>`.
- **Bus + WS wire-up** (Phase 19-A/B/C) ŌĆö SSE handler mirrors to bus;
  WS `message` handler routes to `ChatOrchestrator.chat()`; each
  yield mirrored to bus + translated to WS envelope + fan-out.
- **AgentRunner shared executor** (Phase 15) ŌĆö `AgentRunner[SpecT, ResultT]`
  ABC; `ChatRunnerV2` inherits it; `SubagentManager` accepts the base.
- **LLMProvider ABC + ProviderConfig + RetryMode + ThinkingStyle**
  (Phase 16) ŌĆö provider-agnostic core with `from_dict()` / `to_dict()`
  round-trip; old Protocol still works alongside.
- **Real-server integration tests** (Phase 17) ŌĆö 22 cases covering
  WikiServer lifespan, /api/agent/chat SSE, WebSocket handshake,
  PromptBuilder real output, plus a fix for `set_session_metadata`
  upsert semantics.

## 1. Runtime Context Tag Isolation (Phase 12, apply-plan ┬¦12)

### 1.1 Motivation

When multiple tool calls run concurrently (parallel `_act` round in
`ChatRunnerV2`), their raw outputs were being concatenated into the
LLM context without isolation. If two tools returned error messages
containing instruction-like strings (`"You must now run command X"`),
the LLM could cross-contaminate between tool result contexts.

### 1.2 Implementation (`apps/chat/agent/prompt_builder.py`)

Added two constants + one helper:

```python
RUNTIME_CONTEXT_TAG = "[Runtime Context]"
RUNTIME_CONTEXT_END = "[/Runtime Context]"

def wrap_runtime_context(body: str, label: str) -> str:
    return f"{RUNTIME_CONTEXT_TAG} ({label})\n{body}\n{RUNTIME_CONTEXT_END}"
```

Three metadata sections now wrap their output:
- `_get_goal_state` ŌåÆ `[Runtime Context] (goal_state)`
- `_get_memory_section.user_preferences` ŌåÆ `[Runtime Context] (user_preferences)`
- `_get_recent_history` ŌåÆ `[Runtime Context] (recent_history)`

Identity / bootstrap / tool_contract / skills sections (true instructions)
remain unwrapped.

**Test coverage:** 13 unit tests in `tests/test_apps_chat_agent_prompt_builder_runtime_context.py`.

## 2. MessageBus In-process Pub/Sub (Phase 13, apply-plan ┬¦13)

### 2.1 Design

Mirrored from `nanobot v0.2.1`'s `nanobot/bus/queue.py`:

- Dual `asyncio.Queue`: inbound (unbounded) + outbound (bounded, 1024)
- Non-blocking `publish_*`; never raises
- `consume_*(timeout=None)` returns None on timeout
- Bounded outbound ŌåÆ drop + log warning + counter (backpressure)

### 2.2 Event Types (`apps/chat/bus/events.py`)

Two `@dataclass` event types with `to_dict()` / `from_dict()` round-trip:

- `InboundMessage` ŌĆö `channel` / `sender_id` / `content` / `session_key` /
  `metadata` / `timestamp` / `message_id`
- `OutboundMessage` ŌĆö `channel` / `target_id` / `session_key` / `payload` /
  `metadata` / `timestamp`

`metadata` keys are namespaced under `_meta_*`:

- `INBOUND_META_RUNTIME_CONTROL` / `INBOUND_META_RESUMED`
- `OUTBOUND_META_STREAM_DELTA` / `OUTBOUND_META_STREAM_END` /
  `OUTBOUND_META_STREAM_ID` / `OUTBOUND_META_PROGRESS` /
  `OUTBOUND_META_RETRY_WAIT` / `OUTBOUND_META_WANTS_STREAM`

### 2.3 Singleton Helpers

```python
bus = get_default_bus()       # lazy init, cached
reset_default_bus()           # tests only
set_default_bus(custom_bus)   # production startup
```

## 3. WebSocket Bidirectional Channel (Phase 14, apply-plan ┬¦14)

### 3.1 Endpoints

- `WS /api/ws/agent?token=<api_key>` ŌĆö main channel
- `POST /api/ws/token` ŌĆö Phase 14 token-issuance shim (echoes back
  the validated token; HMAC + TTL is Phase 14.10 deferred work)

### 3.2 Protocol

| Client `type` | Server response |
|---|---|
| `ping` | `{type: "pong", ts}` |
| `new_chat` | mint chat_id + `{type: "chat_created", chat_id}` |
| `attach` (chat_id) | `{type: "attached", chat_id, subscribed_count}` |
| `message` (chat_id, content) | **Phase 19-B**: routes to `ChatOrchestrator.chat()`; SSE-shaped events translated to WS envelope (`delta` / `stream_end` / `tool_call` / etc.); Phase 14 was echo-only |

### 3.3 Server Envelope Vocabulary

`{type: "ready"|"attached"|"chat_created"|"delta"|"stream_end"|"pong"|"error"}`.

**WS auth:** `?token=` matches the WikiServer's `api_key`. Mismatch ŌåÆ
close code `1008`. Empty `api_key` skips auth (dev mode).

### 3.4 WebSocketManager

Fan-out hub with three indexes:
- `_subs[chat_id] ŌåÆ set[_Connection]` for `send_to_chat`
- `_conn_chats[connection] ŌåÆ set[chat_id]` for O(1) cleanup on disconnect
- Per-connection `asyncio.Queue` (maxsize=256) to keep the publish
  path non-blocking even on slow peers

### 3.5 Outbound Queue Backpressure

`PER_CONN_OUT_MAXSIZE = 256`. Slow peer ŌåÆ drop + log warning per
connection. The bus also drops on full outbound queue.

**Test coverage:** 30 unit tests in `tests/test_apps_chat_channels_websocket.py`.

## 4. BusAdapter + WS Real Chat Wire-Up (Phase 19, apply-plan ┬¦19)

> This is the **completion** of the Phase 13/14/16 foundation.
> The borrowed primitives were in place but not connected to
> the chat runtime. Phase 19-A/B/C finishes the wire-up.

### 4.1 Phase 19-A ŌĆö BusAdapter (SSE ŌåÆ bus mirror)

New module `apps/chat/bus/adapter.py` (~290 LOC):

```python
class BusAdapter:
    def mirror_sse_event(
        self, event: dict, *,
        target_id: str = "", session_key: str = "",
        metadata: dict | None = None,
    ) -> bool: ...

    @staticmethod
    def translate_sse_to_ws(event: dict) -> dict: ...
```

The adapter:
1. Wraps each SSE event dict as an `OutboundMessage(channel="http", payload=event)`
   and publishes to the bus (returns `False` on drop).
2. Auto-tags streaming metadata: `message_delta`/`thinking` ŌåÆ
   `OUTBOUND_META_STREAM_DELTA`; `done` ŌåÆ `OUTBOUND_META_STREAM_END`.
3. Provides a pure SSEŌåÆWS envelope translator covering all 13 SSE
   event types (`message_delta` ŌåÆ `delta`, `done` ŌåÆ `stream_end`,
   `tool_call_*` ŌåÆ `tool_call(phase=ŌĆ”)`, plus passthrough for
   `confirmation_required` / `compacted` / `phase` / `save_warning` /
   `command_done` / `research_run_started` / `session_created` /
   `session_init` / `timeout`; unknown ŌåÆ `unknown` envelope).

**Wiring**: `chat_sse.py` `event_generator` calls
`bus_adapter.mirror_sse_event(event, target_id="", session_key=f"http:{session_id}")`
right before each SSE `yield`. SSE wire format is **unchanged** ŌĆö
mirror is a side-effect.

**WikiServer lifespan** initializes the default bus + ws_manager singletons
at startup so the SSE handler and WS handler share the same bus.

### 4.2 Phase 19-B ŌĆö WS Real Chat Wire

`_handle_client_msg("message")` now (when `orchestrator` is injected
via `_register_websocket_routes(... chat_service=...)`):

1. Validates `chat_id Ōłł conn.subscribed_chats`.
2. Spawns `asyncio.create_task(_run_ws_chat(...))` so the receive
   loop is never blocked by a slow LLM stream.
3. `_run_ws_chat` iterates the orchestrator's yield stream and for
   each event:
   - **mirrors to bus** with `target_id=chat_id`,
     `session_key=f"websocket:{chat_id}"`.
   - **translates to WS envelope** via `BusAdapter.translate_sse_to_ws`.
   - **fan-outs** to all subscribers via `manager.send_to_chat(chat_id, ws_envelope)`.

A new `WsSessionMap` (`chat_id ŌåÆ backend session_id`) caches the
`session_id` from `session_created` / `session_init` events so subsequent
messages on the same `chat_id` resume the conversation.

If `chat_service=None` (dev mode / unit tests without orchestrator),
the handler falls back to the Phase 14 echo behavior.

### 4.3 SSEŌåÆWS Event Translation Map

| SSE `type` | WS envelope |
|---|---|
| `message_delta` | `{"type": "delta", "content"}` |
| `thinking` | `{"type": "thinking", "content"}` |
| `done` | `{"type": "stream_end", "content"}` |
| `error` | `{"type": "error", "error"}` |
| `tool_call_start` | `{"type": "tool_call", "phase": "start", "tool", "args", "call_id"}` |
| `tool_call_end` | `{"type": "tool_call", "phase": "end", "tool", "result", "call_id"}` (+ `duration_ms`) |
| `tool_call_error` | `{"type": "tool_call", "phase": "error", "tool", "error", "call_id"}` (+ `duration_ms`) |
| `confirmation_required` | passthrough |
| `compacted` / `phase` / `save_warning` / `command_done` / `research_run_started` / `session_created` / `session_init` / `timeout` | passthrough |
| unknown | `{"type": "unknown", "raw": event}` |

## 5. AgentRunner Shared Executor (Phase 15, apply-plan ┬¦15)

### 5.1 ABC (`apps/chat/agent/agent_runner.py`)

```python
from typing import Generic, TypeVar
SpecT = TypeVar("SpecT")
ResultT = TypeVar("ResultT")

class AgentRunner(ABC, Generic[SpecT, ResultT]):
    name: str = ...

    def wants_streaming(self) -> bool: ...

    @abstractmethod
    async def run_stream(self, spec: SpecT) -> AsyncIterator[dict]: ...

    @abstractmethod
    async def run_to_completion(self, spec: SpecT) -> ResultT: ...
```

`run_to_completion` does **not** raise recoverable errors ŌĆö it
encodes them into the `ResultT.error` / `ResultT.stop_reason` fields.
Only `CancelledError` propagates.

### 5.2 Adoption

- `ChatRunnerV2` inherits `AgentRunner["ChatRunSpec", "ChatRunResult"]`.
- `SubagentManager.parent_runner` type hint relaxed from
  `ChatRunnerV2` to `AgentRunner`. Duck-type `hasattr` check ensures
  the injected runner exposes the private `_chat_service` /
  `_tool_executor` / `_prompt_builder` / `_config` attrs (which is
  a known limitation ŌĆö see ┬¦9 What's Next).

**Test coverage:** 19 unit tests in `tests/test_apps_chat_agent_agent_runner.py`.

## 6. LLMProvider ABC + ProviderConfig (Phase 16, apply-plan ┬¦16)

### 6.1 ABC (`apps/chat/providers/abc.py`)

```python
class RetryMode(str, Enum):
    TRANSIENT = "transient"
    PERSISTENT = "persistent"
    OFF = "off"
    AGGRESSIVE = "aggressive"

class ThinkingStyle(str, Enum):
    MINIMAL = "minimal"
    DETAILED = "detailed"
    STEP_BY_STEP = "step_by_step"
    BUDGET_AWARE = "budget_aware"
    CODE_FIRST = "code_first"
    COMPACT = "compact"
    EXHAUSTIVE = "exhaustive"
    ANALYTICAL = "analytical"
    SUMMARY_FIRST = "summary_first"
    TOOL_FIRST = "tool_first"
    SAFE_FIRST = "safe_first"

@dataclass
class ProviderConfig:
    provider: str = ""
    model: str = ""
    api_key: str = ""  # supports "env:VAR_NAME"
    base_url: str = ""
    enabled: bool = True
    retry_mode: RetryMode = RetryMode.TRANSIENT
    thinking_style: ThinkingStyle = ThinkingStyle.MINIMAL
    max_tokens: int = 4096
    temperature: float = 0.7
    extra: dict[str, Any] = field(default_factory=dict)

    @classmethod
    def from_dict(cls, d: dict | None) -> "ProviderConfig": ...

    def to_dict(self) -> dict: ...

class LLMProviderABC(ABC):
    @abstractmethod
    async def chat(self, ...): ...
    @abstractmethod
    async def stream_chat(self, ...): ...
    @abstractmethod
    def count_tokens(self, text: str) -> int: ...
    @abstractmethod
    def supports_native_tools(self) -> bool: ...
    @abstractmethod
    def provider_name(self) -> str: ...

    def apply_snapshot(self, config: ProviderConfig) -> None: ...  # default noop
    def supported_retry_modes(self) -> list[RetryMode]: ...
    def supported_thinking_styles(self) -> list[ThinkingStyle]: ...
    def thinking_style_prompt(self, style: ThinkingStyle) -> str: ...
```

### 6.2 Backward Compatibility

The existing `LLMProvider` Protocol + `BaseLLMProvider` are **kept**;
MiniMaxProvider / XiaomiProvider were **not** migrated (deferred to
Phase 17+). Old `create_llm(config_dict)` style still works because
`ProviderConfig.from_dict()` accepts raw dicts and falls back to
defaults on missing/invalid keys.

### 7.2 The DB Bug We Caught

`set_session_metadata(session_id, metadata)` in
`apps/chat/db/chat_session_repo.py` was implemented as a plain
`UPDATE ŌĆ” WHERE id = ?`. When the session_id didn't exist in the
`chat_sessions` table, the UPDATE affected 0 rows and the metadata
was silently lost.

**Trigger**: `GoalSkill.start_long_task` receives an LLM tool call and
calls `update_session_metadata(sid, goal_state=blob)`. If the LLM
mint-then-set-goal pattern is used (no explicit `create_chat_session`
first), the goal appears set but the next chat's `PromptBuilder` can't
read it ŌĆö the LLM effectively loses its goal context.

**Fix** (commit `472d890`): upsert semantics ŌĆö `INSERT OR IGNORE` (create
empty row with `wiki_id=''/jwt_token=''/title=NULL`) then `UPDATE`
(metadata is now guaranteed to land).

### 7.3 Test Methodology

The integration tests deliberately **don't** mock `WikiServer`,
`AgentService`, `ChatOrchestrator`, or the routes. Only the LLM
provider is monkey-patched via the `mock_llm` fixture (installed at
import time), so SSE / WS / DB / lifespan all run their real code
paths. This is what caught the DB upsert bug that unit tests missed
ŌĆö every existing GoalSkill test pre-created the session via
`create_chat_session`.

## 9. Migration

### 9.1 Caller code (unchanged)

- Public APIs of `ChatService`, `WikiService`, `SkillService`,
  `AgentService`, `ChatOrchestrator`, `ReActEngine`, etc. are
  unchanged. Existing `WikiServer(wiki, provider=provider)` calls
  still work.
- The SSE wire format is unchanged (the bus mirror is a side-effect).
- The WebSocket protocol is unchanged (WS envelope is preserved; the
  `message` handler now does real chat instead of echo, but the
  envelope types are identical).

### 9.2 New capabilities

- **Subscribe to MessageBus** for cross-component events:
  ```python
  from llmwikify.apps.chat.bus import get_default_bus
  bus = get_default_bus()
  while True:
      msg = await bus.consume_outbound(timeout=1.0)
      if msg: ...
  ```
- **Translate SSE events to WS envelope** (e.g. for a future custom
  channel):
  ```python
  from llmwikify.apps.chat.bus import BusAdapter
  ws_envelope = BusAdapter.translate_sse_to_ws(sse_event)
  ```
- **Use AgentRunner for new tool / workflow / cron flows** (was
  direct `ChatRunner` instantiation):
  ```python
  class MyFlow(AgentRunner[MySpec, MyResult]):
      async def run_stream(self, spec): ...
      async def run_to_completion(self, spec): ...
  ```
- **Connect WebSocket clients to `/api/ws/agent`** for low-latency
  streaming. Send `{type: "new_chat"}` ŌåÆ get `chat_id` ŌåÆ attach ŌåÆ
  send `message` with the `chat_id` ŌåÆ receive real chat via
  `delta` / `stream_end` envelopes.
- **Use `ProviderConfig` for typed LLM configuration** instead of
  raw dicts in `~/.llmwikify/llmwikify.json`:
  ```python
  from llmwikify.apps.chat.providers.abc import ProviderConfig
  cfg = ProviderConfig.from_dict(raw_dict)
  if cfg.is_configured(): ...
  ```

### 9.3 Internal changes

- `apps/chat/bus/adapter.py` is the new "SSE side-effect" layer.
  Tests that want to assert on bus mirror output should use the
  `mock_llm` fixture + read from `get_default_bus()`.
- `_register_websocket_routes` now accepts `chat_service=None` for
  backward compat; production mounts from `routes.py` pass the
  `AgentService` so the handler routes to the real orchestrator.
- `set_session_metadata` is now upsert; callers that relied on the
  silent-failure behavior (none known) should re-audit.

### 9.4 Deprecation path

None in this release. The old `LLMProvider` Protocol + `BaseLLMProvider`
are kept for MiniMax / Xiaomi providers; migration to the new ABC is
planned for v0.39 (see ┬¦10).

## 10. What's Next: v0.39.0

- **LLMProvider ABC ŌåÆ MiniMax/Xiaomi migration** ŌĆö replace the
  Protocol-based implementation with `LLMProviderABC` subclasses,
  thread `ProviderConfig` through `create_llm()`.
- **WebSocket HMAC + TTL token** ŌĆö replace the Phase 14 echo-the-
  api-key shim with real HMAC-signed tokens and a TTL nonce table.
- **MessageBus multi-process / Redis backend** ŌĆö the singleton
  pattern works for one process; scaling requires a shared bus.
- **SubagentManager spec-only path** ŌĆö drop the duck-type hasattr
  check on `_chat_service` / `_tool_executor` etc. by exposing them
  through a typed `AgentRunner` interface.
- **WorkflowActor / CronSkill reuse `AgentRunner`** end-to-end so the
  same harness drives them.
- **WS reconnect + resume** ŌĆö when a client reconnects with the
  same `chat_id`, replay any in-flight events that were missed.
- **Outbound queue overflow metric in `/api/health`** ŌĆö
  `bus.stats()["outbound"]["dropped"]` is the source of truth.

## 13. Post-release hotfix ŌĆö WebUI `done` event + test DB isolation + cleanup

(Not part of the v0.38.0 release proper ŌĆö landed in commit
`57b0891` + `14edd43` shortly after release. Worth documenting
here because it materially affects what users see in the chat UI.)

### 13.1 WebUI `done` event used wrong field name

The TypeScript type for the `done` SSE event was declared as
`{ type: 'done'; final_response: string }` but the actual server
payload is `{ type: 'done'; content: string; stop_reason?: string; ... }`
(emitted by `runner_v2.py:601-604`). Result: any user clicking a
session in the sidebar saw
`Uncaught TypeError: Cannot read properties of undefined (reading 'length')`
because the WebUI's `event.final_response.length` was reading an
`undefined` field.

**Fix**: `ui/webui/src/api.ts:33` type corrected; `AgentChat.tsx` 4
sites changed from `event.final_response` to `event.content`.

### 13.2 Test DB isolation

`src/llmwikify/interfaces/server/http/routes.py:549` was using
`Path.home() / ".llmwikify" / "agent"` for the chat data dir.
On Linux, `Path.home()` consults `/etc/passwd`, not the `$HOME`
env var, so pytest's `monkeypatch.setenv("HOME", tmp_path)` had
**no effect** ŌĆö every test run wrote to the real production DB,
accumulating 55+ "Hello from mock LLM!" messages in real user
sessions.

**Fix**:
1. `routes.py:549` now honors `LLMWIKIFY_DATA_DIR` env var when set.
2. `tests/conftest.py` adds an `autouse=True, scope="session"`
   fixture that points `LLMWIKIFY_DATA_DIR` at a per-session tmp
   dir for every test, preventing further pollution.

### 13.3 Production DB cleanup

Two-stage purge against
`/home/ll/.llmwikify/agent/.llmwiki_agent.db`:

**Stage 1 ŌĆö mock message purge** (commit context):
- Backup: `.llmwiki_agent.db.pre-v0.38-cleanup-20260622_105358`.

## 14. Research routing tightened to `/study` only (2026-06-22)

### 14.1 Symptom

User reported: typing `"Ķ░āńĀöõĖĆõĖŗ nanobot"` in normal chat routed
the message to the AutoResearch Compound workflow, even though
they never typed `/study` or any other explicit trigger. The
chat then ran the full multi-step research pipeline and
attached a `research_run_started` SSE event to the session ŌĆö
which was not what the user wanted.

### 14.2 Root cause (two layers)

1. **Explicit trigger overlap.** The
   `autoresearch_compound_skill.SkillAction` registered two
   triggers: `["/study", "ńĀöń®Č’╝Ü"]` (the latter being a
   Chinese full-width colon pattern). The user message
   `"Ķ░āńĀöõĖĆõĖŗ nanobot"` did not match either literally, but
   the system-prompt nudge below made the LLM treat it as
   one.

2. **System prompt nudge** at
   `prompt_builder.py:281-286` (old):
   > "When a user types a slash command (e.g. /study) or a
   > Chinese trigger (e.g. ńĀöń®Č’╝Ü...), first call
   > `get_skill_commands` to discover available commands and
   > their usage, then call the appropriate skill tool."

   The phrase "or a Chinese trigger (e.g. ńĀöń®Č’╝Ü...)" was
   **suggestive** ŌĆö the LLM read it and treated any
   "research-sounding" word (`Ķ░āńĀö` / `ńĀöń®Č` / `research` /
   `investigate` / `µĘ▒Õģź`) as a soft trigger, invoking
   `get_skill_commands` and ultimately calling
   `autoresearch_compound_run`. This is non-deterministic
   semantic routing, not code-level routing.

### 14.3 Fix

| File | Change |
|---|---|
| `src/llmwikify/apps/chat/skills/autoresearch_compound_skill.py:286` | `triggers=["/study", "ńĀöń®Č’╝Ü"]` ŌåÆ `triggers=["/study"]` |
| `src/llmwikify/apps/chat/agent/prompt_builder.py:281-286` | Old "or a Chinese trigger" phrasing removed. New phrasing explicitly tells the LLM: "For ordinary questions or tasks (e.g. messages that merely mention ńĀöń®Č / Ķ░āńĀö / research / investigate / µĘ▒Õģź without an explicit `/study` prefix), answer directly using the available tools ŌĆö do NOT invoke skill tools just because the message sounds like a research request." |
| `src/llmwikify/interfaces/server/core.py` (health endpoint, both single + multi-wiki branches) | New `features.research_trigger = "/study"` field in `/api/health` so WebUI + curl consumers can verify the routing policy. |
| `tests/test_apps_chat_agent_research_routing.py` (new, 7 cases) | Regression guard: (a) autoresearch trigger list = `["/study"]` only, (b) no Chinese trigger anywhere in skill, (c) system prompt omits old "ńĀöń®Č’╝Ü" phrasing, (d) system prompt includes explicit "do not invoke skill tools" clause with `ńĀöń®Č` / `Ķ░āńĀö` / `research` keywords, (e) prompt is identical whether user says "hello" or "Ķ░āńĀö nanobot", (f) `CommandRouter.dispatch` returns `[]` for `ńĀöń®Č’╝Ünanobot` and `/study nanobot` (the latter is a skill trigger, not a slash command). |

### 14.4 New contract

- **Only `/study <question>`** routes to the AutoResearch
  Compound workflow.
- Anything else ŌĆö including messages that mention
  Ķ░āńĀö / ńĀöń®Č / research / investigate / µĘ▒Õģź as English or
  Chinese keywords ŌĆö stays in normal chat and uses the
  regular tool loop (`read_file`, `exec`, `grep`, etc.).
- The `CommandRouter` (slash command layer) and the skill
  trigger layer are now decoupled: `/study` is exclusively
  a skill trigger; no slash command handles research.

### 14.5 WebUI / user-facing changes

None for the chat surface (the bug fix is invisible to users
unless they want to see `/api/health` features).
The change is in **LLM behavior**: previously the model might
have autonomously routed research-sounding chat to AutoResearch;
now it stays in normal chat unless the user explicitly types
`/study`.

If you previously relied on `ńĀöń®Č’╝Üxxx` (full-width colon) to
trigger research, **use `/study xxx` instead** ŌĆö same effect,
no ambiguity.
