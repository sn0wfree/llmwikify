## What's New in v0.32.5

v0.32.5 is a **focused two-part release** that builds on v0.32.0:

1. **Skill pipeline split (Phase 12a-d)** — extracts `gather_skill` /
   `report_skill` from `research_skill.py`, adds 4 CRUD skills
   (memory/notify/scheduler/dream), and a 28-action `wiki_query_skill`
   aggregator.

2. **3-facade database refactor (Stage A0-E)** — splits the
   11-table `ChatDatabase` into 3 facades (`ChatDatabase`,
   `ResearchDatabase`, `WikiDatabase`) over one physical
   `.llmwiki_agent.db` file, with a new `AppDatabase` aggregate
   for service injection.

The big idea: **databases and skills are both broken into
focused, testable units.** Each facade owns a single domain
(chat, research, wiki-ops); each skill owns a single
capability. v0.33.0 will wire these into the 5+1-service
architecture (`ChatService` + `WikiService` + `SkillService`
+ `HarnessService` + `MemoryManager` + `AgentService`).

## 1. Skill Pipeline Split (Phase 12)

### 1.1 `gather_skill` + `report_skill` (Phase 12a)

Extracted from `research_skill.py` into their own Skill classes
in `apps/chat/skills/`. `research_skill.py` now composes them
as `gather` + `report` phases of the 7-step ReAct pipeline.

- `gather_skill.py` — multi-source fetcher (web, search, file)
- `report_skill.py` — markdown report assembler
- 23 new tests

### 1.2 CRUD Skills (Phase 12b)

Four new skills for the wiki-ops domain, currently registered
in `SkillRegistry` but not yet wired into any service
(dead code, pending v0.33.0 service injection):

- `memory_skill` — ConversationStore, KnowledgeStore,
  ContextStore, ReActStateStore, UserPreferenceStore,
  MemoryIndex
- `notify_skill` — NotificationManager
- `scheduler_skill` — WikiScheduler
- `dream_skill` — DreamEditor
- 36 new tests

### 1.3 `wiki_query_skill` (Phase 12c)

A 28-action aggregator that unifies the three tool surfaces
(`WikiToolRegistry` 21 tools, `WikiQuerySkill` 28 actions,
MCP 29 functions). Currently runs in parallel with the others;
will become the single entry point in v0.33.0.

- 17 new tests

### 1.4 Async + Test Isolation (Phase 12d)

- Enabled `asyncio_mode=auto` in `pytest.ini` (154 async tests
  unblocked)
- Fixed mock test isolation (deprecation warnings in `WikiService`

## 2. 3-Facade Database Refactor (Stage A0-E)

### 2.1 `autoresearch.db` → `.llmwiki_agent.db` (Stage A0)

Renamed the canonical DB filename. Old `autoresearch.db`
files are silently auto-migrated on first access.

### 2.2 `BaseDatabase` (Stage A)

New `apps/db_base.py:BaseDatabase` provides shared lifecycle:

- `__init__(data_dir)`: resolves canonical path, calls
  subclass `_init_db()`, checks db size
- `get_app_db_path(data_dir)`: canonical path helper
  (auto-migrates `autoresearch.db` → `.llmwiki_agent.db`)
- `_init_db()`: subclass-overridable (each facade creates
  only its own tables)
- `_check_db_size()`: warn if > 100 MB

### 2.3 `ChatDatabase` (Stage A + C)

Now inherits `BaseDatabase`. Shrinks to **4 tables**:
- `chat_sessions`, `chat_messages`, `tool_calls`
- `context_entries` (NEW, for v0.33.0 MemoryManager)

The 17 wiki-domain methods and 27 research-domain methods
are thin delegates (via lazy `self._wiki` and
`self._research` properties) to `WikiDatabase` and
`ResearchDatabase` respectively.

**`AutoResearchDatabase = ChatDatabase` (subclass)** — old
imports still work; this subclass ensures all 3 facades
initialize on construction (backward compat).

### 2.4 `ResearchDatabase` (Stage B)

New `apps/research/db.py:ResearchDatabase` (727 lines,
27 methods). Owns 4 research tables:
- `autoresearch_sessions`, `autoresearch_sub_queries`,
  `autoresearch_sources`, `research_steps`

`apps/research/{session,engine,gatherer,analyzer}.py` and
`apps/chat/skills/research_skill.py` updated to use
`ResearchDatabase` instead of `ChatDatabase`.

### 2.5 `WikiDatabase` (Stage A)

New `apps/wiki/db.py:WikiDatabase` (403 lines, 17 methods).
Owns 4 wiki-ops tables:
- `dream_proposals`, `notifications`, `confirmations`,
  `ingest_log`

### 2.6 `AppDatabase` (Stage D)

New `apps/db.py:AppDatabase` (56 lines) aggregates the
3 facades into one injection point:

```python
from llmwikify.apps.db import AppDatabase

db = AppDatabase(data_dir)
db.chat.create_chat_session(wiki_id="w1")
db.research.create_research_session(wiki_id="w1", query="q")
db.wiki.save_notification({"wiki_id": "w1", ...})
```

### 2.7 `AgentService` accepts `AppDatabase` (Stage E)

`AgentService` wrapper updated to accept an optional
`app_db: AppDatabase` parameter. If not provided, it
constructs one internally. Backward compat preserved.

## 5. Migration Notes

### 5.1 DB filename

`autoresearch.db` → `.llmwiki_agent.db`. The old file is
auto-renamed on first access (silent, no data loss).
No user action required.

### 5.2 Import paths (backward compat preserved)

| Old import | New import | Status |
|---|---|---|
| `from llmwikify.apps.chat.db import ChatDatabase` | Same | ✅ |
| `from llmwikify.apps.agent.core.db import AgentDatabase` | `from llmwikify.apps.chat.db import ChatDatabase` | ⚠️ Deprecated, removed in v0.33.0 |
| `from llmwikify.apps.agent.core.service import AgentService` | `from llmwikify.apps.chat.agent.service import ChatService` | ⚠️ Deprecated, wrapper kept |
| `from llmwikify.apps.autoresearch.engine import ResearchEngine` | `from llmwikify.apps.research.engine import ResearchEngine` | ✅ |
| `from llmwikify.apps.autoresearch.session import ResearchSessionManager` | `from llmwikify.apps.research.session import ResearchSessionManager` | ✅ |

### 5.3 New code style

Prefer `AppDatabase` over separate facades:

```python
# Old (v0.32.0)
from llmwikify.apps.chat.db import ChatDatabase
from llmwikify.apps.wiki.service import WikiService

db = ChatDatabase(data_dir)
wiki_svc = WikiService(registry, data_dir, db)

# New (v0.32.5)
from llmwikify.apps.db import AppDatabase
from llmwikify.apps.wiki.service import WikiService

app_db = AppDatabase(data_dir)
wiki_svc = WikiService(registry, data_dir, app_db.chat)
```

## 6. Known Issues (4 pre-existing failures)

- `test_pdf_missing_dep_returns_error` — pymupdf 1.26+ rejects
  fake PDF content (test wrote `"fake pdf content"` to a
  `.pdf` file)

- `test_existing_pdf_extraction_with_pymupdf` — same pymupdf issue
- `test_preserves_youtube_routing` —
  `youtube_transcript_api` >= 1.0 removed `get_transcript()`
  (renamed to `fetch()`)

These are 3rd-party library compatibility issues, not v0.32.5
regressions. CI may pass in an isolated environment with
older library versions.

## 7. What's Next: v0.33.0

v0.32.5 sets up the database and skill infrastructure.
v0.33.0 will wire them into the 5+1-service architecture:

1. `SkillService` (new facade) — populates `SkillContext.config`
   so CRUD skills work in production
2. `HarnessService` (new facade) — unifies 5 eval primitives
   (QualityGate, SourceFilter, etc.)
3. `MemoryManager` (new module) — 6 stores (Conversation,
   Knowledge, Context, ReActState, UserPreference, MemoryIndex)
4. `AgentService` rewrite as composition root (delete wrapper)
5. Delete `agent/core/{db,service}.py` wrappers

Estimated effort: ~14h across 8 stages (per
`docs/designs/v0.33-service-refactor.md`).
