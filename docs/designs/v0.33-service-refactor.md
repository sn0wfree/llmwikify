> **Status:** Designed, ready to implement
> **Branch:** `feat/v0.32-skill-refactor` (continuation)
> **Estimated effort:** ~14 hours across 8 stages
> **Date:** 2026-06-08

## 1. Motivation

The v0.32.0 / v0.32.5 refactor created two production services — `ChatService`
(apps/chat/agent/service.py) and `WikiService` (apps/wiki/service.py) — plus
two deprecated wrappers (`AgentService`, `AgentDatabase`). The two-service
model is **insufficient** for the long-term architecture:

1. **3 parallel tool surfaces** — `WikiToolRegistry` (21 tools), `WikiQuerySkill`
   (28 actions), and MCP `@mcp.tool` (29 functions) duplicate ~80% of their
   parameter schemas. None of them is a single source of truth.

2. **5+ evaluation primitives constructed ad-hoc** — `QualityGate`,
   `SourceFilter`, `StructureValidator`, `ResearchReviewer`, `ResearchRevisor`,
   `SourceAnalyzer` are instantiated at ~30 call sites with no shared config.

3. **Skill framework disconnected from services** — 30 distinct Skill classes
   (23 base + 4 CRUD + 2 pipelines + 1 aggregator) live in
   `apps/chat/skills/` but **no service** populates `SkillContext.config` for
   them. CRUD skills are dead code in production.

4. **Triple ReAct loop** — `ReactLoop` (framework), `chat/engine.py` (6-step),
   `apps/research/engine.py` (7-step) coexist. The framework version was
   designed to replace the other two but only `research_skill.py` uses it.

5. **Triple DB name** — `ChatDatabase` + `AgentDatabase` (wrapper) +
   `AutoResearchDatabase` (alias) all point to the same SQLite file.

6. **DB facade has 11 tables across 3 domains** in one class:
   - chat (sessions, messages, tool_calls)
   - research (autoresearch_sessions/sub_queries/sources, research_steps)
   - wiki-ops (dream_proposals, notifications, confirmations, ingest_log)

The fix is a **5+1-service architecture** with **3 database facades** over
a single physical SQLite file, and a **`MemoryManager` memory base**
sitting parallel to the database layer.

## 3. The 3-Database-Facade Architecture

### 3.1 One physical database, three logical facades

```
Physical file: ~/.llmwikify/agent/.llmwiki_agent.db (renamed from autoresearch.db)
                12 tables in one SQLite file

ChatDatabase (apps/chat/db.py)            WikiDatabase (apps/wiki/db.py) [NEW]      ResearchDatabase (apps/research/db.py) [NEW]
  ├── chat_sessions                          ├── dream_proposals                       ├── autoresearch_sessions
  ├── chat_messages                          ├── notifications                          ├── autoresearch_sub_queries
  ├── tool_calls                             ├── confirmations                          ├── autoresearch_sources
  └── context_entries [NEW]                 └── ingest_log                             └── research_steps
```

### 3.2 Why one physical file, three facades

| Reason | Benefit |
|---|---|
| Cross-table atomicity | `tool_calls` ↔ `chat_sessions` ↔ `research_sessions` need referential integrity |
| One backup file | `cp .llmwiki_agent.db backup.db` |
| No cross-database JOINs | No `ATTACH DATABASE` complexity |
| 3 facades give domain clarity | Each facade exposes only its own tables |

### 3.3 `BaseDatabase` (shared parent)

```python
# apps/db_base.py

class BaseDatabase:
    """Shared SQLite lifecycle for all 3 facades.

    Auto-migrates autoresearch.db → .llmwiki_agent.db on first
    construction (backward-compat for existing installations).
    """
    def __init__(self, data_dir: Path | str):
        self.data_dir = Path(data_dir)
        self.db_path = self._resolve_db_path()
        self._init_db()  # 3 facade-specific _init_db calls are merged here
        self._check_db_size()

    def _resolve_db_path(self) -> Path:
        # 1. If .llmwiki_agent.db exists, use it
        # 2. If autoresearch.db exists, rename to .llmwiki_agent.db
        # 3. Otherwise create .llmwiki_agent.db
        new_path = self.data_dir / ".llmwiki_agent.db"
        old_path = self.data_dir / "autoresearch.db"
        if old_path.exists() and not new_path.exists():
            old_path.rename(new_path)
        return new_path

    def _connect(self) -> sqlite3.Connection:

        ...
```

### 3.4 `AppDatabase` (3-facade aggregate)

```python
# apps/db.py

class AppDatabase:
    """Aggregate of 3 database facades over one physical file.

    Each facade exposes only its domain's tables. Service
    injection is one parameter, not three.
    """
    def __init__(self, data_dir: Path | str):
        self.data_dir = Path(data_dir)
        self.chat = ChatDatabase(data_dir)
        self.wiki = WikiDatabase(data_dir)
        self.research = ResearchDatabase(data_dir)
```

### 3.5 `AgentService` composition

```python
# apps/chat/agent/service.py

class AgentService:
    """Composition of ChatService + SkillService + HarnessService
    + MemoryManager + WikiService.

    Replaces the deprecated apps/agent/core/service.py::AgentService.
    """
    def __init__(
        self,
        wiki_service: WikiService,
        app_db: AppDatabase,
        skill_service: SkillService | None = None,
        harness_service: HarnessService | None = None,
        memory_manager: MemoryManager | None = None,
    ):
        self.wiki_service = wiki_service
        self.app_db = app_db
        self.chat_service = ChatService(wiki_service, app_db.chat)
        self.skill_service = skill_service or SkillService()
        self.harness_service = harness_service or HarnessService(
            llm_client=wiki_service.get_llm()
        )
        self.memory_manager = memory_manager or MemoryManager(
            app_db=app_db,
            wiki=wiki_service.get_default_wiki(),
            data_dir=app_db.data_dir,
        )

    # Backward-compat delegation for deprecated wrapper callers
    async def chat(self, message, **kw): ...
    def run_dream(self, **kw): ...
    def list_notifications(self, **kw): ...
    # etc. — all 19 original AgentService methods
```

## 4. The 6-Store MemoryManager

`MemoryManager` is a **memory base** that runs parallel to the database
layer. It owns all the "memory-style" data that doesn't fit a relational
schema cleanly.

### 4.1 Store matrix

| Store | Storage | Replaces | API |
|---|---|---|---|
| `ConversationStore` | JSONL | `apps/agent/memory/ConversationMemory` | `append/get_recent/get_all/clear/search` |
| `KnowledgeStore` | `wiki.query_sink` | `apps/agent/memory/SinkMemory` | `append_answer/read/status/pending_pages/search_by_query` |
| `ContextStore` | SQLite (new `context_entries` table) | in-memory `ChatService._contexts` | `save/get/delete/clear` |
| `ReActStateStore` | SQLite (`research_steps` table) | in-memory `ReactLoop` state dict | `save_state/load_state/list_steps` |
| `UserPreferenceStore` | JSON | none (new) | `get_pref/set_pref/list_prefs` |
| `MemoryIndex` | in-memory | none (new) | `search_all(query, stores, limit)` |

### 4.2 `context_entries` table schema

```sql
CREATE TABLE IF NOT EXISTS context_entries (
    id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL,
    key TEXT NOT NULL,
    value TEXT,
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now')),
    UNIQUE (session_id, key),
    FOREIGN KEY (session_id) REFERENCES chat_sessions(id)
)
```

### 4.3 Cross-domain access

`MemoryManager` reads from multiple facades:

```python
class MemoryManager:
    def __init__(self, app_db: AppDatabase, wiki, data_dir):
        self._chat = app_db.chat          # for ConversationStore + ContextStore
        self._research = app_db.research  # for ReActStateStore
        self._wiki = wiki                 # for KnowledgeStore (wiki.query_sink)
```

## 6. Risk Matrix

| Risk | Severity | Mitigation |
|---|---|---|
| Auto-migration breaks user data | 🔴 high | Migration is a simple `os.rename()` — same file content, different name. Tested with backup-then-migrate flow. |
| Cross-domain access in MemoryManager | 🟡 med | Application layer (not facade) does the cross-domain composition. |
| 3-facade split breaks production routes | 🟡 med | Each stage runs full test suite + architecture check before commit. |
| WikiDatabase / ResearchDatabase duplicate methods | 🟡 med | Each facade's methods are scoped to its own domain. No overlap. |
| ResearchEngine merge (Phase 4) not done | 🟢 low | Not in this plan. The two ReAct engines coexist; research_skill.py is the preferred path. |
| Backward-compat with `AgentService` wrapper | 🟢 low | Wrapper is deprecated; we keep it as a thin facade for 1-2 versions. |
