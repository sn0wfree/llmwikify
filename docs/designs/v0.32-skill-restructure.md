# v0.32 Skill 重构 Design Doc

> **Sprint v0.32 of the 4-layer refactor series.**
> Per-session discussions 2026-06-08 distilled into a single actionable design document.
> Supersedes the `apps/agent/` deprecation mark from v0.30 and the 5901 LOC agent/ + 3442 LOC research/ + 3038 LOC ppt/ sprawl.

## 1. 背景与目标

### 1.1 现状问题

| 维度 | 现状数据 |
|---|---|
| `apps/agent/` | 21 文件 **5901 LOC**（WikiAgent runner + 20+ tools + memory + notify + scheduler + providers + hooks + dream_editor + routes）|
| `apps/research/` | 15 文件 **3442 LOC**（6 步研究子模块独立存在）|
| `apps/ppt/` | 10 文件 **3038 LOC**（PPT 生成独立存在）|
| `apps/chat/` | 31 文件 **7851 LOC**（6 步研究引擎 + chat 框架 + harness 三合一）|
| `apps/` 总占比 | **51.9%**（20251 / 38994 LOC）|

**核心问题**：
- **`apps/agent/` 是"什么都干"的巨型包**（同时承载 WikiAgent runner / 20+ tools / memory / notify / scheduler / providers / hooks / dream_editor / routes / db）
- **`apps/chat/` 混杂**（6 步研究子模块 + chat 框架 + eval harness）
- **`apps/research/` 与 `apps/chat/engine.py` 高度重叠**（同 6 步研究逻辑两边各一份）
- **`apps/ppt/` 与 chat 集成**（chat_router/chat_engine/chat_routes 三件套 793 LOC）
- **apps 内 4 个子包关系模糊**（chat 是中心？research/ppt 是被调模块？）

### 1.2 目标

1. **业务域清晰化**：apps 内 5 个业务域（wiki/chat/research/ppt/skills）边界明确
2. **Skill 化**：每个能力单元独立可测、可调、可复用
3. **Unix 哲学**：底层 12 skill = 工具池，上层 9 skill = 业务流程（详见 `docs/designs/principles/unix-philosophy.md`）
4. **Orchestrator = 上层 skill 的别名**：不重复实现
5. **MCP 零破坏**：28 个 wiki_* tool 派生自 wiki_orchestrator skill，外部客户端不受影响
6. **PPT 暂缓**：apps/ppt/ 21 文件保持现状，后续可能重写
7. **Tauri 桌面化**：v0.32 完成后立即启动 v0.33.0
8. **v0.32 分两阶段**：v0.32.0（核心 1.5 月）+ v0.32.5（完整 1 月）

### 1.3 非目标

- 不做 Tauri 桌面化（v0.32 完成后立即做，但不在 v0.32 范围）
- 不做 i18next 多语言
- 不迁移到 Milkdown 编辑器
- 不重写 apps/ppt/（PPT 暂缓，后续可能重写）
- 不完全 DDD 化（采用方案 1 而非方案 2）

## 3. Skill 体系（Unix 哲学）

详见 `docs/designs/principles/unix-philosophy.md`。3 条核心原则：
1. **底层 = 工具**（单一职责、纯数据操作）
2. **上层 = 业务流程**（编排底层完成业务目标）
3. **简单 CRUD 不拆**（已经是 Unix 风格）

### 3.3 上层 skill 调底层 skill 的内部机制

```python
# apps/chat/skills/gather_skill.py
class GatherSkill(Skill):
    name = "gather"
    actions = {
        "gather_for_research": SkillAction(handler=gather_for_research, ...),
    }

async def gather_for_research(args, ctx):
    """业务流程：search → filter → extract."""
    # 1. 调底层 search_skill
    search_result = await search_skill.actions["search"].handler(
        {"query": args["query"], "limit": 50}, ctx
    )

    # 2. 调底层 filter_skill
    filter_result = await filter_skill.actions["filter"].handler(
        {"sources": search_result.data["results"], "min_score": 0.3}, ctx
    )

    # 3. 调底层 extract_skill
    sources = []
    for s in filter_result.data["filtered"][:args.get("max_sources", 20)]:
        extract = await extract_skill.actions["extract"].handler(
            {"url_or_path": s["url"]}, ctx
        )
        sources.append({**s, "content": extract.data["content"]})

    return SkillResult(status="ok", data={"sources": sources})


# apps/chat/skills/research_skill.py
async def run_research(args, ctx):
    """6 步研究：plan → gather → analyze → summarize → score → revise → report."""
    state = ResearchState()

    # 步 1: plan (底层)
    state.plan = await plan_skill.actions["plan"].handler(args, ctx)

    # 步 2: gather (上层, 内部调 search+filter+extract)
    state.sources = await gather_skill.actions["gather_for_research"].handler(
        {"query": args["query"], "sub_queries": state.plan.data["sub_queries"]}, ctx
    )

    # 步 3: analyze (底层)
    state.analyzed = await analyze_skill.actions["analyze"].handler(..., ctx)

    # 步 4: summarize (底层)
    state.summary = await summarize_skill.actions["summarize"].handler(..., ctx)

    # 步 5: score (底层)
    score = await score_skill.actions["score"].handler(..., ctx)

    # 步 5b: revise (底层) 如 score < 阈值
    if score.data["score"] < ctx.config.get("quality_threshold", 7):
        state.summary = await revise_skill.actions["revise"].handler(..., ctx)

    # 步 6: report (上层, 内部调 summarize+write+score)
    final = await report_skill.actions["generate_report"].handler(..., ctx)

    return SkillResult(status="ok", data={"report": final.data, "state": state.to_dict()})
```

**关键设计点**：
- **上层 skill 调底层 skill** 通过 Python 函数调用（**不通过 LLM**）
- 状态机 6 步全部由 Python 驱动
- 状态持久化在 `research_skill.run_research` 内部
- LLM 可选粒度：高层一键 vs 底层组合

## 6. MCP 28 tool 派生（零外部破坏）

### 6.1 派生机制

```python
# interfaces/mcp/tools.py (NEW)
def register_skill_tools(mcp, registry):
    for skill in registry.skills.values():
        for action_name, action in skill.actions.items():
            @mcp.tool
            def {action_name}(**kwargs):
                return action.handler(kwargs, ...)
```

### 6.2 零破坏保证
- 28 个 tool **name 不变**：`read_page`, `write_page`, `search`, ...
- 28 个 tool **描述字符串**：`wiki_query_skill.actions["read_page"].description`（**显式 hardcode 原文**，不从 manifest 重新生成）
- 28 个 tool **参数 schema**：与原一致
- 测试：`assert registered_tool.description == original_description`（string equality）

- read_page / write_page / search / search_semantic

- search_research / list_research_sessions

每个 action 内部调底层 skill（read/write/search/extract/lint/graph/score）。
