# v0.39 — WebSearchSkill & gather_skill web fallback

## Problem

External web search is the LLM's **only** way to fetch current
information that isn't already in the user's wiki. Yet until v0.39:

- `WebSearch` (DuckDuckGo / Tavily / SearXNG / MiniMax) was a
  **library-only** module, accessible only from internal code
  paths (`apps/research/gatherer.py`, `apps/chat/gatherer.py`).
- It was **never** exposed as an LLM-callable tool.
- The LLM had no way to ask "search the web for X" interactively.
- The research gather pipeline could not fall back to the web when
  the local wiki returned nothing.

## Solution (two parts)

### Part 1: New `WebSearchSkill` (24th base Skill, 3 actions)

LLM-callable tool set, thin wrapper over the existing
`apps/research/web_search.py::WebSearch`:

| Action | Purpose |
|---|---|
| `web_search.search_web(query, num_results)` | General web search |
| `web_search.search_youtube(query, num_results)` | `site:youtube.com` routing |
| `web_search.search_news(query, num_results)` | `site:news.google.com` routing |

The provider fallback chain is unchanged (SearXNG → MiniMax →
Tavily → DuckDuckGo). Configuration lives in `ctx.config`:

```python
ctx.config = {
    "search_provider": "auto",          # or "searxng" / "minimax" / "tavily" / "duckduckgo"
    "searxng_url": "http://localhost:8888",
    "minimax_api_key": "...",
    "tavily_api_key": "...",
    "web_search_results_per_query": 5,
}
```

Validation:
- `query`: required, non-empty after stripping
- `num_results`: integer 1-50, default 5
- Provider exceptions → `SkillResult.fail` (LLM sees error message)
- Empty results → `SkillResult.ok({"results": [], "count": 0})`

### Part 2: `gather_skill` opt-in fallback

`gather_for_research` gains an `enable_web_search: bool` arg
(default `False` for backwards compatibility). When True and the
wiki returns zero results for a sub-query, `WebSearch` is invoked
and its results are appended (with `source_type="web"`) before
moving to the next sub-query.

```
Per sub-query:
  ┌─ wiki.search(query) → has results? → use them, done
  └─ enable_web_search?
       ├─ True  → WebSearch.search(query) → append, done
       └─ False → leave sub-query unsourced (existing behavior)
```

Failure modes:
- WebSearch raises → log + swallow, gather still succeeds
- WebSearch URL already in `sources` → skipped (dedup)

## Future work

- **Dedicated news provider** (newsapi.org, Google News RSS) —
  `search_news` currently uses `site:news.google.com` on the same
  fallback chain.
- **Result caching** — `WebSearch` results are not currently
  memoized. Repeated sub-queries re-hit the providers.
- **Provider-specific quotas** — MiniMax's 450 req/day limit is
  global; per-session quota tracking would prevent starvation.
- **Score / filter integration** — `SourceFilter` is applied to
  web-gathered results in the research engine but not yet in
  `gather_skill`; could be added via `enable_source_filter`.
- **`web_search_skill` and `search_skill` consolidation** — both
  are search tools; consider a single `search` Skill with
  sub-actions `wiki`, `web`, `youtube`, `news`.
